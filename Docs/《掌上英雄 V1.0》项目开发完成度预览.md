## 四、Phase 15.5 新增开发规范

> 以下规则由 Phase 15.5 开发过程中确立，适用于后续所有战斗/UI 开发。

### RULE-001: 可选节点安全引用模式

场景树的节点可能不存在（如 HPBar 对非精英怪物不可见），必须使用 null-safe 的 `@onready` 声明：

```gdscript
@onready var hp_bar: TextureProgressBar = null if not has_node("HPBar") else $HPBar
```

使用前增加 `is_instance_valid()` 保护：
```gdscript
if hp_bar and is_instance_valid(hp_bar):
    hp_bar.value = current_hp
```

**不这样做 →** 缺少节点的预制体实例化时报 `Invalid get index` 崩溃。

### RULE-002: 属性计算三层分离模式

所有战斗属性必须通过 `get_final_stats(stat_name)` 统一出口，数据源分三层：

| 层 | 字典 | 类型 | 来源 | 示例 |
|----|------|------|------|------|
| Base | `_config_data` | Dictionary | `init_hero(data)` 传入的 JSON 模板 | `base_atk: 40` |
| Meta | `_meta_bonuses` | Dictionary | 装备/天赋固定值加成 | `"atk": 10` |
| Buff | `_buff_bonuses` | Dictionary | 战斗内临时百分比 Buff | `"atk": 0.15` |

计算公式：`final = (base + meta) * (1 + star_bonus) * (1 + buff)`

**不这样做 →** 属性来源散落在各处导出变量中，Buff 系统接入时需大规模重构。

### RULE-003: 单帧创建限流模式

在 `_process` 外部（如 Tween 回调、信号响应）批量创建 UI 节点时（伤害飘字、粒子），必须使用静态帧计数器限流：

```gdscript
static var _popup_last_frame: int = -1
static var _popup_frame_count: int = 0

func show_damage_text(...) -> void:
    var current_frame := Engine.get_process_frames()
    if current_frame != _popup_last_frame:
        _popup_last_frame = current_frame
        _popup_frame_count = 0
    if _popup_frame_count >= POPUP_MAX_PER_FRAME:
        return
    _popup_frame_count += 1
    if _popup_frame_count > POPUP_OFFSET_THRESHOLD:
        screen_pos += Vector2(randf_range(-30, 30), randf_range(-20, 20))
```

| 参数 | 值 | 含义 |
|------|-----|------|
| `POPUP_OFFSET_THRESHOLD` | 5 | 超过此值启用随机偏移防重叠 |
| `POPUP_MAX_PER_FRAME` | 15 | 硬上限，超过直接丢弃 |

**不这样做 →** 高伤害场景（AOE/连击）单帧生成数十个 Label，性能崩溃且文字完全重叠。
