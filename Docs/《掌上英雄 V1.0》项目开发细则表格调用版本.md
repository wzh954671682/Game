《掌上英雄 V1.0》项目开发大纲
1. 项目综述与核心概念
1.1 游戏基本信息
1.1.1 游戏定位
*   **核心定义**：一款以 5x5 棋盘为阵地，结合了卡牌部署、即时合成与 Roguelike 元素的基础塔防游戏。
*   **核心卖点**：单手操作的爽快感（Portrait 模式）、高频合成的即时策略反馈、针对“死锁”场景的自动救援机制。
*   **视觉风格**：简洁、高对比度的 2D 美术，强调打击反馈（屏幕震动、死亡缩放特效）。

1.1.2 目标平台
*   **移动端优先**：原生支持竖屏（Portrait），操作区域集中在屏幕中下部。
*   **设计分辨率标准**：
    *   **基准尺寸**：1080 x 2160 (9:18 纵横比)。
    *   **拉伸模式**：Canvas Items 模式，Aspect 设置为 Expand。
*   **刘海屏/全面屏适配标准**：
    *   **安全区域 (Safe Area) 获取**：系统启动时通过 `DisplayServer.get_display_safe_area()` 获取设备实际可用矩形范围。
    *   **顶层 UI 容器化**：所有交互 UI（HUD、手牌区、弹窗）必须包裹在专用的 `MarginContainer` 中。脚本根据 Safe Area 自动计算并设置该容器的顶部（避开刘海）和底部（避开手势条）的像素偏移。
    *   **背景延伸原则**：战斗背景图与转场装饰采用“溢出渲染”策略。背景图设计尺寸应略大于基准分辨率（如 1242 x 2688），确保在长屏手机上填满全屏，不留黑边，而核心逻辑（5x5 棋盘）保持在安全区中心。
    *   **锚点策略**：交互按钮严禁使用绝对坐标，统一使用百分比锚点。详情弹窗固定在安全区域的 80% 居中位置，不受刘海遮挡影响。

1.1.3 技术选型
*   **开发引擎**：Godot 4.x。
*   **渲染后端**：Compatibility 渲染模式，确保在各类移动端设备（尤其是中低端安卓设备）上保持稳定 60 帧。
*   **架构模式**：采用“数据驱动（Resource-Based）+ 全局信号总线（Event Bus）”架构，实现 UI、数值逻辑与战斗表现的高度解耦。

1.2 核心循环 (Core Loop)
1.2.1 局内循环（战斗实时流程）
1.  **抽卡与资源管理**：
    *   手牌上限 5 张。系统自动从“隐藏备用队列”中按序填充。
    *   玩家可通过消耗弃牌次数（[X] 按钮）清理不需要的卡牌。
2.  **部署与强制位移**：
    *   玩家将卡牌拖拽至 5x5 棋盘任意格位。
    *   **判定规则**：若目标格位已有怪物，放置英雄时会触发“强制位移”，将怪物向上方格位挤出，并触发英雄的“拦截”状态。
3.  **合成进化**：
    *   将相同 ID 的英雄拖拽至已部署英雄上方，触发合成升级。
    *   英雄星级从 1 星最高提升至 5 星，且合成时可根据已选路径获得分支属性增益。
4.  **关卡推进与反馈**：
    *   英雄自动执行纵向检测并攻击。
    *   消灭波次怪物、击杀精英/Boss。若格位无英雄，怪物会直接撞击城墙扣除 HP。
5.  **死锁救援**：
    *   若场上英雄数量为 0 且手牌区、备用队列均为空，触发 10 秒冷却的自动补给（5张基础角色卡）。

1.2.2 局外循环（长线成长流程）
1.  **结算获取**：
    *   战斗胜利后根据通关星级（受城墙余血比例影响）分配金币、指定英雄碎片。
2.  **英雄等级提升**：
    *   进入英雄养成界面，消耗对应的碎片与金币提升英雄基础等级（上限 30 级）。
    *   **质变逻辑**：每 5 级（5/10/15...30）需要点击特殊节点激活核心天赋加成（数据通过 Resource 表驱动）。
3.  **城墙强化**：
    *   消耗金币永久升级城墙。每级提升基础 HP 值，强化副本容错率。
4.  **阶段性挑战**：
    *   随着养成属性提升，挑战更高难度的副本（从 6 分钟短途副本过渡到 10 分钟精英 Boss 副本）。

2. 战斗系统逻辑 (Battle System)
2.1 战场布局与基础交互
2.1.1 5x5 棋盘格点坐标映射
*   **逻辑坐标系**：定义为 `Vector2i(0,0)` 到 `Vector2i(4,4)` 的二维数组。
*   **屏幕映射公式**：根据 1080x2160 分辨率及安全区域，计算棋盘中心偏移。
    *   `pos.x = screen_center.x + (col - 2) * cell_size`
    *   `pos.y = screen_center.y + (row - 2) * cell_size`
*   **背景融合**：棋盘格位贴图需与动态场景背景图层分离，支持高光（Highlight）提示当前拖拽目标格位。

2.1.2 角色拖拽反馈逻辑
*   **状态切换**：当手指按下（InputEventScreenTouch）卡牌时，卡牌进入“拖拽态”。
*   **视觉增强**：
    *   **缩放**：Tween 动画将卡牌平滑放大至 1.1x。
    *   **倾斜**：根据拖拽速度（Velocity）给卡牌增加左右 ±5° 的动态倾斜，增加手感灵活性。
    *   **层级**：提升 `z_index` 确保卡牌悬浮在所有 UI 和单位之上。

2.1.3 强制位移（Forced Displacement）判定流程
*   **触发条件**：英雄释放于逻辑格位 `(r, c)`，且该格位当前被怪物占据。
*   **挤压规则**：
    1.  检测目标格位，发现怪物实体。
    2.  执行位移：怪物逻辑坐标更新为 `(r-1, c)`（即向上挤出一格）。
    3.  动画实现：怪物通过 `TRANS_QUART` 曲线平滑移动至新位置。
    4.  **连带判定**：若 `(r-1, c)` 已有其他英雄或怪物，新怪物仍执行强制位移并重叠（优先保证英雄放下不落空），随后触发所有重叠怪物的“拦截”逻辑。

2.2 角色战斗行为
2.2.1 纵向攻击检测机制
*   **组件应用**：每个英雄节点内置一个向上的 `RayCast2D`（或 `ShapeCast2D` 以增加判定宽度）。
*   **检测逻辑**：每帧（或固定频率）检测上方列的第一个有效怪物碰撞体。
*   **攻击执行**：检测成功后触发攻击动画 -> 产生伤害信号 -> 怪物血量扣除。
*   **打击停顿 (Hit Stop)**：高伤害攻击命中时，通过 `Engine.time_scale` 短暂（如 0.05s）下调至 0，增强打击反馈感。

2.2.2 全英雄拦截系统（Universal Interception）
*   **核心定义**：所有英雄单位（不限于盾兵）均具备物理拦截能力。
*   **拦截状态机**：
    *   **Idle**：无怪经过，执行默认待机。
    *   **Intercepting**：怪物进入英雄碰撞范围。怪物停止移动速度（Speed = 0），英雄与怪物互相进入“攻击/阻挡”动画循环。
*   **优先级判定**：英雄可同时拦截的怪物数量受属性 `block_count` 控制。当达到上限时，后续怪物理论上可穿过（透行）该英雄格位，直到城墙。

2.2.3 英雄属性动态计算
*   **计算公式**：`Final_Stat = (Base_Stat + Meta_Progression_Bonus) * (1 + In_Battle_Star_Bonus) * (1 + Buff_Card_Bonus)`。
*   **属性同步**：当合成升级或使用效果卡时，通过信号（Signal）即时更新当前场上实体的攻击速度、伤害值及阻挡上限。

2.3 怪物行为与城墙交互
2.3.1 怪物属性与波次导演
*   **基础属性**：`move_speed`（移动速度）、`attack_dmg`（受拦截时伤害）、`wall_dmg`（撞击城墙伤害）。
*   **血量缩放**：副本进度每过 1 分钟，后续生成的怪物基础血量依据 Resource 配置的系数进行百分比线性增长。

2.3.2 城墙受击反馈与屏幕震动
*   **判定判定**：怪物未被拦截到达屏幕底端坐标 -> 触发 `GameEvents.wall_hit(damage)` -> 怪物销毁。
*   **屏幕震动实现**：
    *   监听信号后，`Camera2D` 执行震动。
    *   **参数控制**：震动强度 `intensity = 5.0`，持续时间 `0.2s`，利用 `randf_range` 实现随机位移偏移（Offset）。

2.3.3 死亡动画标准
*   **表现流程**：
    1.  怪物 HP <= 0，禁用所有碰撞（Collision）与逻辑（Process）。
    2.  **Tween 表现**：
        *   `scale` 从 1.0 缩放到 0（带 `TRANS_BACK` 回弹）。
        *   `modulate:a`（透明度）同步降至 0。
        *   `position.y` 微量向上漂移（模拟消散）。
    3.  **信号触发**：动画结束后释放内存 `queue_free()`，并发送 `enemy_died` 全局信号（用于掉落计算）。
3. 卡牌系统与手牌管理 (Card & Hand Management)
3.1 卡牌流转机制
3.1.1 隐藏备用队列 (Hidden Queue) 逻辑
*   **数据结构**：采用 FIFO（先进先出）队列存储。当玩家升级或获得随机奖励时，产生的卡牌（如每级奖励 3 张）不直接进入手牌，而是推入备用队列。
*   **自动填充逻辑**：
    *   系统每帧或在手牌数量变动时执行检测。
    *   若当前手牌数量 < 5 且 备用队列不为空，则自动将队列首位的卡牌弹出（Pop），并通过 Tween 动画“飞入”空余的手牌槽位。
*   **存储上限**：理论无上限，但为了 UI 表现，仅记录总数。

3.1.2 手牌槽位管理与 9+ 提示
*   **槽位布局**：底部横向排列 5 个固定槽位。每张手牌拥有独立的状态机（在槽位中、拖拽中、被选中）。
*   **队列提示 UI**：
    *   在手牌区侧方设置“待命援军”图标。
    *   **显示逻辑**：实时显示备用队列中的卡牌总数。
    *   **9+ 规则**：当总数 ≤ 9 时显示实际数字；当总数 > 9 时，显示为 “9+”，增强玩家对资源富余量的视觉感知。

3.1.3 弃牌逻辑（Discard Logic）
*   **交互设计**：每张手牌槽位上方或侧角悬挂一个半透明的 [X] 按钮。
*   **判定流程**：
    1.  玩家点击 [X]。
    2.  触发该槽位卡牌的“销毁动画”（如碎裂或向下滑出）。
    3.  立即释放该槽位占用。
    4.  触发 3.1.1 中的自动填充逻辑，从备用队列抽取新牌补充。
*   **设计目的**：允许玩家快速过滤当前不需要的效果卡，寻找急需的角色卡进行合成或补位。

3.2 死锁救援机制 (Anti-Lock System)
3.2.1 强制触发判定
*   **监控目标**：系统实时监控三项指标：`field_hero_count` (场上英雄数)、`current_hand_count` (当前手牌数)、`queue_count` (备用队列数)。
*   **死锁定义**：当上述三项指标同时为 0 时（玩家无兵可用、无牌可打、无援军可补），判定为死锁状态。
*   **检测频率**：每 1 秒执行一次全局扫描，或在英雄死亡且备用队列为空时即时触发。

3.2.2 冷却与状态控制
*   **强制 CD**：一旦救援补给被触发，进入 10 秒强制冷却时间（救援 CD）。
*   **CD 期间逻辑**：在 CD 期间即使再次满足死锁条件，也不会重复发牌，防止玩家利用极端空场刷牌。
*   **UI 反馈**：触发时在屏幕中央弹出“补给抵达”飘字提示（Toast）。

3.2.3 救援补给包内容
*   **发牌策略**：强制向手牌槽位填充 5 张卡牌。
*   **卡池限定**：仅限“基础角色卡池”（如盾兵、机枪手），严禁出现“效果提升卡”或“全局/传说卡”，确保玩家获得的是即时战力而非增益插件。

3.3 卡牌分类与功能定义
3.3.1 角色卡 (Character Cards)
*   **基础要素**：包含 Hero_ID、品质颜色底版、初始星级（1星）。
*   **部署逻辑**：拖拽至棋盘，若该位置已有同 ID 英雄，执行合成逻辑；若位置为空或有怪，执行部署/拦截逻辑。

3.3.2 效果提升卡 (Upgrade Cards)
*   **流派锁定机制**：针对特定角色设计。例如“加特林改装卡”，施加给机枪手后，该机枪手将从“射程流”和“攻速流”中锁定后续合成路径。
*   **叠加限制**：每名英雄战中最多接受 5 次此类强化。

3.3.3 全局与传说卡 (Global/Legendary Cards)
*   **全局效果**：无需选定格位，释放即对全场生效（如：全场敌方冰冻 3s、城墙 5s 免疫伤害）。
*   **传说特技**：具备独特的视觉特效，如屏幕色调反转、时间流速减缓等。
4. 战中合成与分支进化 (In-Battle Progression)
4.1 合成升级逻辑
4.1.1 星级提升规则与数值映射
*   **合成操作**：玩家将手牌中的角色卡拖拽至棋盘上**相同 ID** 且**星级未满**的英雄实体上释放。
*   **星级上限**：最高 5 星。
*   **数值成长公式**：
    *   基础属性（攻击力、生命值、防御力）随星级呈指数或大幅线性增长。
    *   参考映射：2星（1.5x 基础）、3星（2.2x 基础）、4星（3.0x 基础）、5星（4.5x 基础 + 质变特效）。
*   **继承规则**：合成后，被投喂英雄将继承原实体的所有 Buff 状态，并触发“升级回血”效果（恢复 20% 最大生命值）。

4.1.2 视觉进化表现标准
*   **体型缩放**：每提升 1 星，英雄 Sprite 整体缩放比例增加 10%。5 星英雄体型为初始的 1.4x。
*   **光环反馈**：
    *   1-3 星：脚底显示淡色呼吸灯光环。
    *   4 星：光环颜色加深，并伴随粒子向上漂浮。
    *   5 星（满星）：触发“流光溢彩”Shader 效果，角色全身边缘高亮，且脚底光环转变为复杂的旋转符文阵列。
*   **Tween 动画**：合成瞬间，角色执行一个快速的 `upscale` (从 1.0 弹至 1.3 再回到当前星级比例) 的回弹效果。

4.2 战斗分支路径 (Implicit Branching)
4.2.1 隐性分支锁定逻辑
*   **触发时机**：当 1 星英雄首次被施加“效果提升卡（Upgrade Cards）”或与带有特定路径属性的辅助卡合成时。
*   **状态锁定**：一旦确定路径（如选择了“攻速强化”），该英雄实体将永久打上“路径 A”标签。
*   **视觉区分**：
    *   路径 A（攻速/防御）：光环偏向蓝色/绿色调。
    *   路径 B（爆发/反伤）：光环偏向红色/橙色调。
*   **不可逆性**：单局战斗内，该位置的英雄一旦锁定分支，后续只能继续投喂相同分支的强化效果或普通角色卡。

4.2.2 路线 A：攻速/防御流（侧重持续作战）
*   **成长偏向**：每级额外提升 15% 攻击速度或 10% 伤害减免。
*   **5 星质变效果定义（根据角色 ID 不同）**：
    *   **机枪手（A 路径）**：开启“无限火力”，攻击不再有短暂间隔，子弹附带穿透效果（可穿透 2 个敌人）。
    *   **盾兵（A 路径）**：开启“反重力力场”，拦截范围内所有怪物的移动速度额外降低 50%，且自身防御力翻倍。
*   **战斗定位**：阵地战核心，负责稳住战线，对抗大规模尸潮。

4.2.3 路线 B：爆发/反伤流（侧重瞬间清场）
*   **成长偏向**：每级额外提升 20% 暴击率或 25% 反震伤害。
*   **5 星质变效果定义（根据角色 ID 不同）**：
    *   **机枪手（B 路径）**：开启“核聚变弹头”，每第 4 次攻击触发范围爆炸，对 1.5 格范围内敌人造成 300% 伤害。
    *   **盾兵（B 路径）**：开启“荆棘圣盾”，受击时将 50% 的伤害以纯粹伤害形式回馈给拦截对象，且死亡时产生大范围自爆。
*   **战斗定位**：精锐击杀者，负责快速清理精英怪物或在防线崩溃前通过高爆发挽回战局。
5. 局外养成系统 (Meta-Progression)
5.1 英雄 30 级养成线
5.1.1 碎片与金币消耗公式逻辑
*   **消耗资源**：提升英雄等级需同时消耗“英雄专属碎片”与“通用金币”。
*   **碎片增长公式**：`Cost_Shards = Base_Shard_Value + (Current_Level / 5) * Step_Multiplier`。碎片需求随等级段阶梯式上升。
*   **金币增长公式**：采用非线性增长，确保后期金币消耗具有压力。
*   **等级上限**：V1.0 版本最高等级设定为 30 级。

5.1.2 5 级间隔质变节点设计
*   **节点分布**：在等级达到 5、10、15、20、25、30 时，解锁特殊的“核心天赋”节点。
*   **节点解锁判定**：
    *   玩家等级达到对应数值。
    *   必须在英雄详情面板中手动点击激活节点。
    *   激活节点可能需要额外的特殊资源或大量金币。
*   **功能定义范例**：
    *   **10级节点**：基础攻击速度永久提升 15%。
    *   **25级节点**：攻击附带微量减速或弹射效果。
*   **UI 表现**：质变节点在详情页以特殊的六边形图标显示，未解锁时为灰色，解锁后高亮并带有呼吸灯特效。

5.1.3 基础属性持久化增长
*   **成长映射**：每一级（非节点等级）提升都会为英雄带来 ATK（攻击）、HP（生命）、DEF（防御）的百分比提升。
*   **数值同步**：局外养成的最终数值将作为战斗开始时的 `Base_Stat` 进入 2.2.3 的计算公式。

5.2 城墙养成体系
5.2.1 城墙等级与 HP 加成
*   **成长逻辑**：城墙等级提升主要增加战斗容错率。
*   **HP 加成公式**：`Wall_HP = Base_HP * (1 + Current_Level * Growth_Coefficient)`。
*   **系数定义**：假设每级提升 15% 基础血量。

5.2.2 升级金币消耗算法
*   **公式定义**：`Upgrade_Cost = upgrade_base_cost * pow(current_level, cost_exponent)`。
*   **指数系数**：推荐 `cost_exponent` 设定在 1.2 - 1.5 之间，使高等级城墙升级成本极高，引导玩家平衡英雄与城墙的投入。

5.3 奖励与掉落逻辑
5.3.1 副本结算奖励分配
*   **金币收益**：根据关卡难度与击杀怪物数量计算基础收益，受星级倍率加成。
*   **碎片掉落**：
    *   固定奖励：首次通关奖励指定英雄碎片。
    *   随机掉落：概率从已拥有的英雄池中抽取碎片，确保玩家能够持续提升已有战力。

5.3.2 三星挑战目标体系
*   **1 星（基本完成）**：在规定时间内消灭最终 Boss 通关。
*   **2 星（精锐防御）**：通关且城墙剩余血量比例 > 80%。
*   **3 星（完美无缺）**：通关且城墙血量保持 100%（未受伤害）。
*   **结算反馈**：三星通关将额外奖励 50% 的金币产出，并解锁该关卡的“扫荡”或“快速挑战”功能（后续扩展）。
6. UI/UX 界面与交互 (Interface Design)
6.1 主框架与导航
6.1.1 五 Tab 底部导航设计
*   **核心导航栏**：屏幕最下方固定高度区域，包含五个核心入口：[商城]、[英雄]、[战斗]、[营地]、[公会]。
*   **状态反馈**：当前选中 Tab 需有明显的视觉偏移或高亮动效（如图标上移 10 像素并改变颜色）。
*   **适配处理**：导航栏底部需根据 `DisplayServer.get_display_safe_area()` 预留系统手势条避让空间（约 100 像素）。

6.1.2 英雄界面交互逻辑
*   **上阵槽位**：界面顶部展示 5 个空余/已填充的英雄槽位。
*   **角色列表**：下方以 GridContainer 展示玩家拥有的所有英雄卡牌。
*   **部署操作**：玩家长按列表中的卡牌并拖拽至上方槽位完成“上阵”。
*   **品质视觉**：卡牌底版根据英雄品质（R/SR/SSR）显示对应的边框特效及背景色块。
*   **交互详情**：单点列表中的英雄卡牌，触发 6.2.1 定义的英雄详情弹窗。

6.1.3 战斗首页（关卡选择）
*   **Boss 预览**：以旋转轮播图形式展示当前关卡的最终 Boss 模型及基础属性描述。
*   **难度标识**：分为“普通”、“困难”、“地狱”三个等级，通过不同颜色的字体和光效区分。
*   **星级预览**：显示当前关卡最高完成星级，点击星级图标弹出具体挑战目标（如“城墙血量 100%”）。

6.2 弹窗与交互细节
6.2.1 通用弹窗基类 (BasePopup) 设计标准
*   **视觉覆盖**：背景为 `#000000B2`（70% 不透明度）的半透明黑色 ColorRect，负责阻挡点击并营造深度感。
*   **内容区域**：锚点设为中心 80% 区域（`Margins: 0.1, 0.1, 0.9, 0.9`）。
*   **进入动画**：
    *   **Scale 缩放**：从 `(0.8, 0.8)` 平滑过渡到 `(1.0, 1.0)`。
    *   **Modulate 透明度**：从 `0` 渐变至 `1.0`。
    *   **曲线类型**：使用 `Tween.TRANS_BACK` 配合 `Tween.EASE_OUT` 实现轻微的回弹效果。
    *   **Pivot**：必须在 `_ready` 中动态设置 `pivot_offset = size / 2` 确保从中心点缩放。

6.2.2 栈式弹窗管理逻辑
*   **遮罩点击关闭**：监听背景 ColorRect 的 `gui_input` 事件。若检测到点击，调用关闭函数。
*   **层级锁定**：当弹窗开启时，通过 `set_process_input(false)` 或遮罩层阻断下方场景（如棋盘、列表）的任何交互。
*   **关闭流程**：先执行缩放缩小动画，动画结束后调用 `queue_free()`。

6.2.3 结算界面奖励弹出流程
*   **背景入场**：全屏渐暗，中心出现“胜利/失败”标题。
*   **奖励列表生成**：根据结算数据，预先实例化奖励图标（碎片、金币、装备）。
*   **依次弹出逻辑**：
    *   使用 `await` 或 `create_timer` 配合循环。
    *   **时间间隔**：每隔 `0.15s` 弹出一个奖励图标。
    *   **弹出动效**：单个图标从 `scale = 0` 快速放大至 `scale = 1.0`，带回弹曲线。
    *   **音效配合**：每个图标弹出时伴随短促的清脆音效。

6.3 战斗反馈 UI
6.3.1 伤害飘字 (Damage Floating Text) 系统
*   **普通伤害**：
    *   **颜色**：白色字体，黑色描边。
    *   **动效**：匀速垂直上漂 80 像素，伴随透明度渐隐。
*   **暴击伤害**：
    *   **颜色**：黄色/橙色字体，字体大小为普通的 1.5 倍。
    *   **动效**：初始瞬间执行一个快速的 `Pop-up` 缩放动画（0 -> 1.2 -> 1.0），然后缓慢上漂。
*   **生命周期**：飘字实例生存时间固定为 `0.6s`，动画结束后立即销毁。

6.3.2 动态血条显示逻辑
*   **普通怪物**：不显示常驻血条，受击时不弹出（保持画面整洁）。
*   **精英/Boss 怪物**：
    *   **显示机制**：受击瞬间在头顶显示 `TextureProgressBar`。
    *   **Boss 特化**：Boss 拥有位于屏幕顶部的常驻分层血条，受击时血条有“残影扣除”效果（主血条先扣，白影随后平滑滑向目标值）。
    *   **消失逻辑**：怪物死亡或一段时间未受击后，血条渐隐消失。

6.3.3 系统反馈 Toast
*   **死锁补给提示**：当触发“救援机制”时，屏幕中央弹出带背景底板的文字提示“补给抵达：获得基础援军”，停留 `1.5s` 后渐隐。
*   **错误码提示**：碎片不足、金币不足等操作错误，根据错误码表映射文案，以红色文字在屏幕中间轻微跳动显示。
7. 数值表格与技术架构 (Technical Design)
7.1 全表驱动配置 (Resource-Based)
7.1.1 关卡配置表 (LevelResource)
*   **核心字段**：
    *   `level_id`: String (关卡唯一标识)
    *   `difficulty_coefficient`: float (全局数值缩放系数)
    *   `wave_data`: Array[WaveResource] (波次详情，含怪物类型、生成间隔、路径偏差)
    *   `bg_texture`: Texture2D (背景贴图路径)
    *   `grid_tile_set`: TileSet (5x5 棋盘对应的格位贴图)
    *   `reward_pool`: Dictionary (首次通关奖励及概率掉落池)

7.1.2 英雄成长表 (HeroProgressionResource)
*   **基础字段**：`hero_id`, `base_atk`, `base_hp`, `base_def`, `block_count`
*   **等级数据**：
    *   `level_costs`: Dictionary (Key: Level, Value: {gold: int, shard: int})
    *   `growth_curve`: Curve (各等级属性成长曲线)
*   **节点配置**：
    *   `node_data`: Array[UpgradeNodeResource] (存储 5/10/15...30 级的质变天赋，含图标、文案及对应的逻辑修饰符 ID)

7.1.3 错误码表 (ErrorCodeResource)
*   **映射机制**：定义一个全局的 `Dictionary[int, String]`。
*   **范例内容**：
    *   `100`: "金币不足，快去战斗获取吧！"
    *   `101`: "碎片不足，无法升级该英雄。"
    *   `102`: "城墙已达到最高等级。"
    *   `200`: "角色已满级。"
*   **触发流程**：逻辑脚本抛出错误 ID -> Toast 系统根据 ID 检索表 -> 弹出动画显示文案。

7.2 程序底层架构
7.2.1 全局信号总线 (GameEvents.gd)
*   **定义方式**：单例 (Autoload)，负责全局事件分发。
*   **核心信号清单**：
    *   `wall_hit(damage: int, current_hp: int)`: 城墙受击，触发屏幕震动及 UI 更新。
    *   `enemy_died(global_pos: Vector2, enemy_type: String)`: 敌人死亡，触发飘字及掉落逻辑。
    *   `card_deployed(logic_pos: Vector2i, hero_data: HeroResource)`: 卡牌成功放置。
    *   `deadlock_rescue_triggered()`: 救援机制开启，触发 UI 播报。
    *   `currency_changed(type: String, new_amount: int)`: 金币/碎片变动同步 UI。

7.2.2 数据持久化管理 (SaveManager.gd)
*   **存储方案**：使用 Godot `ConfigFile` 类。
*   **加密策略**：
    *   **接口**：`FileAccess.open_encrypted_with_pass()`。
    *   **密钥**：自定义固定密钥字符串，防止玩家通过文本编辑器直接修改本地金币。
*   **存档路径**：`user://save_v1.dat`。
*   **存储内容**：英雄等级字典、关卡进度（星级）、城墙等级、全局资源（金币/碎片总数）。

7.2.3 场景管理与异步切换
*   **加载流程**：
    1.  从主界面点击 [战斗开始]。
    2.  加载界面 (Loading Scene) 启动。
    3.  异步加载战斗资源（使用 `ResourceLoader.load_threaded_request`）。
    4.  加载完成后，执行 `change_scene_to_packed` 切换。
*   **状态保持**：通过全局单例 `GameManager` 暂存当前关卡数据，确保战斗结束返回主界面时能正确恢复 Tab 状态。

7.3 性能优化标准 (Performance Standards)
*   **对象池 (Object Pooling)**：
    *   对高频出现的 [子弹]、[伤害飘字]、[基础小怪] 实施对象池管理。
    *   死亡时不执行 `queue_free()`，而是标记为无效并隐藏，待下次生成时复用。
*   **检测优化**：英雄纵向攻击检测频率由每帧改为每 0.1s 执行一次，降低 CPU 物理射线检测负担。
*   **内存管理**：详情弹窗关闭后强制触发垃圾回收逻辑或确保所有 Tween 信号已断开引用，防止内存泄漏。
