# 《掌上英雄 V1.0》面向 AI 代理 (Agent-friendly) 开发流程与里程碑

## 核心开发原则
*   **数据驱动**：所有数值通过 Resource 或 JSON 调整，严禁在脚本里写死。
*   **信号解耦**：UI 不直接控制战斗逻辑，统一走 `GameEvents.gd` 信号总线。
*   **强类型约束**：所有 GDScript 必须使用强类型声明（`: Type`）以减少运行时错误。
*   **Agent 友好**：任务拆解至微型颗粒度，确保 AI 代理（如 Claude Code）单次对话可闭环完成。

## Phase 1: 基础设施与数据契约 (Foundation)
**目标**：建立项目全局单例与核心算法，定义 AI 后续调用的 API 契约。

- [ ] **Task 1.1: 全局信号总线 (Event Bus)**
    - **内容**：创建 `src/Global/GameEvents.gd` 单例。
    - **QA 自查**：参数是否强类型？信号命名是否符合下划线规范？
- [ ] **Task 1.2: JSON 数据加载器 (Data Manager)**
    - **内容**：创建 `src/Global/DataManager.gd`。实现安全读取 `Data/*.json` 的方法。
    - **QA 自查**：是否包含 `file_exists` 检查？`JSON.parse_string` 结果是否判定 `null`？
- [ ] **Task 1.3: 棋盘坐标映射器 (Grid Logic)**
    - **内容**：创建 `src/Logic/GridManager.gd`。实现 5x5 `Vector2i` 与屏幕像素 `Vector2` 双向转换。
    - **QA 自查**：是否考虑了 Safe Area 偏移？是否对越界坐标做了 `clamp` 或警告？

## Phase 2: 战斗实体与核心判定 (Battle MVP)
**目标**：实现“英雄拦截”与“怪物位移”核心卖点逻辑。

- [ ] **Task 2.1: 怪物基础移动 (Enemy Base)**
    - **内容**：实现 `src/Entities/Enemy.gd`。受击减速与到达底线触发 `wall_hit`。
    - **QA 自查**：是否使用了 `_physics_process`？信号触发是否传参？
- [ ] **Task 2.2: 英雄拦截系统 (Hero Interception)**
    - **内容**：实现 `src/Entities/Hero.gd`。集成 `RayCast2D` 进行纵向检测，实现拦截状态切换。
    - **QA 自查**：拦截数是否受 `block_count` 限制？检测频率是否优化为 0.1s？
- [ ] **Task 2.3: 强制位移算法 (Forced Displacement)**
    - **内容**：在 `BattleManager` 中实现英雄放下时的挤压逻辑。
    - **QA 自查**：是否使用了 `Tween.TRANS_QUART`？是否处理了连续挤压的递归？

## Phase 3: 卡牌系统与手牌逻辑 (Card Logic)
**目标**：实现手牌管理、隐藏队列与死锁补给。

- [ ] **Task 3.1: 手牌槽位管理 (Hand Manager)**
    - **内容**：管理 5 个手牌实例的布局、拖拽状态切换。
    - **QA 自查**：拖拽时 `z_index` 是否提升？是否正确捕获 `gui_input`？
- [ ] **Task 3.2: 备用队列与死锁救援 (Queue & Anti-Lock)**
    - **内容**：实现 FIFO 补牌逻辑与 10s CD 的全空救援补给。
    - **QA 自查**：补牌是否有 0.1s 间隔 `await`？救援触发是否包含 5 秒开局静默期？

## Phase 4: UI 表现与持久化 (UX & Meta)
**目标**：处理屏幕适配、打击感反馈与加密存档。

- [ ] **Task 4.1: 刘海屏 Safe Area 适配**
    - **内容**：编写全局适配脚本，动态设置 UI 容器的 Margin。
    - **QA 自查**：是否使用了 `DisplayServer.get_display_safe_area()`？
- [ ] **Task 4.2: 打击感增强 (VFX)**
    - **内容**：实现屏幕震动与 Hit Stop（打击停顿）。
    - **QA 自查**：`Engine.time_scale` 调节后是否能在 0.05s 后正确恢复？
- [ ] **Task 4.3: 加密存档系统 (Save Manager)**
    - **内容**：使用 `ConfigFile` 加密存储金币与关卡进度。
    - **QA 自查**：密钥是否硬编码？文件读写是否异常容错？

### 为什么这样排期？

1.  **先解决“位移”和“拦截”**：这是你游戏区别于普通塔防的最核心逻辑。如果在 Phase 1 发现 5x5 挤压怪物的感觉不对，我们还能调，如果等到 UI 做好了再改逻辑，工作量是翻倍的。
2.  **异步 UI 开发**：Phase 3 集中处理 UI，是因为那时候你的 `GameEvents` 信号已经非常丰富了，UI 只需要作为“信号监听者”去播放动画，完全不影响战斗逻辑。
3.  **持久化放最后**：在开发过程中，我们可以直接在代码里 mock（模拟）数据。存档系统往往会因为系统增加而频繁重构，所以放在最后做最稳妥。
